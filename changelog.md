- 2026-04-12 — Full fix on the Task 18 follow-up bugs (commit `15fa25b3`). Three bugs fixed in one commit after the post-Task-18 fanout made the failure modes concrete: (1) `jobs/_registry.js` was passing `{ teamSize: 4, teamConcurrency: 4 }` to `boss.work()`, but those are pg-boss **v9** options — in v10 the only valid fetch option is `batchSize` (default 1), so the v9 names were silently ignored and every queue ran a single-worker fetch loop. That's why a slow `ingest-topic-from-source` job blocked every sibling in the same fanout during Task 18. Fix: new `concurrency` option in the `register()` helper that calls `boss.work()` N times to create N independent workers for the same queue, each with its own fetch cycle. ingest-topic-from-source now has 4 parallel workers. (2) The same helper had a latent silent-data-loss hazard: `const jobRow = Array.isArray(job) ? job[0] : job;` — in v10 the work callback ALWAYS receives an array of job rows, so this dropped every job after the first if `batchSize > 1`. Fixed by iterating the batch via `Promise.all()` and returning per-job results. Not currently reachable (effective batchSize was 1) but pinned via regression tests so it won't silently regress. (3) biorxiv/medrxiv/sportrxiv client-side scans could walk all 12 monthly chunks (biorxiv/medrxiv) or the entire OSF preprint pool (sportrxiv) even when the topic yielded zero matches, which is pointless and slow. Added newest→oldest scan direction, `MAX_CONSECUTIVE_EMPTY_CHUNKS=3` early-exit on biorxiv/medrxiv, `MAX_PAGES_PER_CHUNK=5` per-chunk safety cap, and `MAX_PAGES=10` hard cap on sportrxiv's flat pagination. Added `tests/unit/jobs/_registry.test.js` (6 new tests pinning the concurrency contract + batch-iteration behavior) and a biorxiv early-termination regression test. **Validation**: re-ran the Task 18 fanout against topic 1 with target=20. `core` completed in 6 seconds (15 inserted + 5 skipped) WHILE `biorxiv` was still running its scan — the smoking-gun proof that the concurrency fix works.
